# CLAUDE.md — ツーリングVlog自動生成プロジェクト

## プロジェクト概要

バイクのマスツーリング（グループでのツーリング）の走行映像から、AIがハイライトを自動検出して約30分のダイジェスト動画を生成するツール。個人の趣味プロジェクト。

詳細はインセプションデッキを参照: [docs/inception-deck.md](docs/inception-deck.md)

---

## ユーザーについて

- **開発者＝エンドユーザー本人**。自分のツーリング動画を自分で使うために作っている
- バイクのマスツーリング愛好者で、仲間とのツーリングの思い出を動画で残している
- 編集の手間を省きたいのが動機

---

## 出力動画の用途

- YouTubeの**限定公開（URLを知っている人だけが見られる）** で仲間に共有
- 公開（一般公開）は想定していない
- 尺は約30分

## 機材と素材について

### カメラ構成（1回のツーリングで同時使用）

| 機材 | 用途 | 備考 |
|---|---|---|
| EDR-22G（ドラレコ） | 前方・後方カメラ映像 + GPS記録 | 30秒セグメントで複数ファイル生成 |
| Insta360 X3 | 360°映像 | .insvファイル（デュアル魚眼）、要スティッチ |
| B+COM 6XR（インカム） | ライダー間の会話音声 | Insta360 X3とBluetooth直接接続で同時録音可能 |

### 典型的な素材量

- 1回のツーリング = 約6時間
- 6時間フル録画前提での実測ベース見積もり（2026/07/03テストツーリングの実ファイルから算出）：
  - EDR-22G（前+後合計、常時録画）：約21.6Mbps ≈ **約58GB/6時間**
  - Insta360 X3（デュアル魚眼2レンズ合計、6時間フル録画した場合）：約80Mbps ≈ **約208GB/6時間**
  - 合計：**約266GB/6時間**（両カメラとも6時間フル録画した場合）
  - 参考：実際のテストツーリングではInsta360は行程中の一部区間（合計62分）のみ録画しており、その場合の実データ量は約37.5GBだった

## 実装済みツール

### `tools/sync_ui.py` — カメラ時刻同期UI（完成）

EDR-22G と Insta360 X3 の時刻オフセットを手動で特定するためのローカルWebUI（Python HTTPサーバー、port 8080）。

- 同一時刻の映像フレームを EDR（左）/ Insta360（右）で3行並べて表示
- ±1秒 / ±10秒 / ±1分 ボタンでそれぞれ調整、ギャップ秒数をリアルタイム表示
- Insta360 のレンズ切替（_00_ 前方 / _10_ 後方）対応

**2026/07/03 テスト走行の結果：** Insta360 RTC は GPS 実時刻より **439秒進んでいる**（Insta360 RTC が速い）。  
EDR は GPS 時刻同期で高精度、Insta360 は RTC のため分レベルのずれが生じる。  
導出方法：sync_ui.py（INSV_START をファイル名タイムスタンプ 09:26:22 に修正後）で gap = **+439** を計測。  
フレーム照合：Insta360 clip009 開始 49 秒（RTC 09:27:12）の実時刻 = 09:27:12 − 439 = 09:19:53 = EDR GPS 時刻、EDR 焼込み 09:19:52 と一致。  
`INSTA360_OFFSET_SEC = -439`（実時刻 = RTC − 439）。  
過去の 430 秒（遅れ方向・逆符号）および 267 秒（clip007 照合・誤り）は誤った値だった。  
オフセット値はツーリングごとに YAML 設定に保存する設計（未実装）。

### `tools/transcribe.py` — 音声文字起こし（完成）

Insta360 X3 の音声（インカム経由）を faster-whisper で文字起こしし、SRT + TXT を出力。

- モデル: `large-v3`（日本語インカム音声で最高精度）
- デバイス: CUDA（RTX 5060 GPU）/ float16
- 出力先: `cam-data/transcripts/large_v3/`

**環境上の注意：** ctranslate2 は CUDA 12.x の `cublas64_12.dll` を要求する。  
CUDA 13.3 が入っていても CUDA 12.9 を**併存インストール**する必要がある（PATH に 12.9 の bin が入ればよい）。

**kotoba-whisper-v2.1 との比較結果：** large-v3 が 418 行に対し kotoba は 16 行。  
kotoba はクリーンな音声向けに最適化されており、ノイジーなインカム音声には large-v3 が大幅に優秀。

### `tools/02_extract_gps.py` — GPS抽出 + インタラクティブマップ（完成）

EDR-22G の全セグメントから GPS を抽出し、タイムスタンプ付き CSV とインタラクティブ HTML マップを生成。

**2026/07/03 テスト走行の実績：**
- 115 セグメント、3030 GPS ポイント
- 時刻範囲: 09:21〜20:19（657.9 分）

**タイムスタンプ割り当て方式：**  
YWKJGPS レコードには時刻フィールドがないため、  
`セグメント開始時刻 + file_fraction × 30.5秒` で推算。

**マップUI仕様（`cam-data/gps/tour_20260703.html`）：**
- タイル: CartoDB Voyager（APIキー不要）
- ポリライン: 青系 `#1552b8`、通過済み weight 8.75 / 未通過 weight 6.25 opacity 0.55
- 時刻スライダーで再生位置をスクラブ、通過済み（濃）/未通過（薄）をリアルタイム更新
- アスペクト比トグル（1:1 / 16:9）、ズームトグル（全体 / 追尾 zoom 17）
- 逆ジオコーディング（Nominatim API）で現在地住所を取得 → テキストエリアからコピー可能
- 動画へのスーパーインポーズを前提とした設計（手法は未実装）

### マップのスーパーインポーズと冒頭サマリCG

**方針確定：Playwright（地図フレーム生成）+ Remotion（冒頭CG）+ FFmpeg（合成）の三層構成。**

#### 役割分担

```
[地図・CG生成]           →  [動画への合成]   →  [エンコード]
 Playwright / Remotion       FFmpeg overlay      FFmpeg
```

#### 走行中の地図スーパーインポーズ

- `tour_20260703.html` は既にスーパーインポーズを前提とした設計（アスペクト比トグル・追尾ズーム対応済み）
- **Playwright** でブラウザを自動操作し、GPS時刻ごとにスライダーを動かしてPNGフレームシーケンスを生成
- 生成したフレーム列を FFmpeg `overlay` フィルタで走行映像の右下などに合成

```python
# 概念コード
await page.evaluate(f"setMapTime({timestamp_sec})")
await page.screenshot(path=f"map_frames/{frame_i:06d}.png")
```

```bash
ffmpeg -i driving.mp4 -r 30 -i map_frames/%06d.png \
  -filter_complex "[1:v]scale=480:270[map];[0:v][map]overlay=W-w-20:H-h-20" \
  output.mp4
```

#### 冒頭サマリCG（コース全体マップ・日付・所要時間・走行距離）

- **Remotion**（React ベース）でCGを記述し、ヘッドレスレンダリング
- HTML/CSS/React で書けるため、`tour_20260703.html` のマップコードを直接流用できる
- CSSアニメーションでルートトレースのアニメーション等も実装可能

```bash
npx remotion render TourSummary summary.mp4
```

最終的に FFmpeg `concat` で冒頭CG + ダイジェスト本編をつなぐ。

## 設計上の判断と背景

### 手動編集機能は作らない

ユーザーがAIの出力を気に入らないときは「テキストで指示」して再生成する設計。編集UIを作ると複雑になりすぎる。

### カメラ選択はAIが話題から自動判断

前カメラ・後カメラ・360°の3ソースから、会話の話題や文脈をもとにAIが使用カメラを選択する。例：「あの交差点」→ 前カメラ、「みんなで休憩中」→ 360°など。

### 仲間のトラッキングにはテキスト入力を使う

仲間のバイクの車種・車体色・服の色をあらかじめテキストで登録し、それを元にオブジェクト検出でトラッキングする。ユーザーに明示的に入力させることで精度を上げる。

### レンダリングは夜間バッチ処理

30分動画のレンダリングは時間がかかる前提で、インタラクティブな操作（分析・区間選定）と重い処理（レンダリング）を分離して設計する。
